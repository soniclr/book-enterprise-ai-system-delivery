# 附录 C：版本化技术卡

核查日：2026-07-17。正文讲稳定决策，本附录记录会随版本变化的事实。读者实施前应重新打开官方来源；核查日期不是长期有效承诺。

## 技术卡 1：MCP 授权与工具安全

| 字段 | 核查结论 |
|---|---|
| 对象 | Model Context Protocol（MCP）授权与工具接入 |
| 已核查版本 | 协议规范 `2025-11-25` 的 Authorization 章节；官方 Security Best Practices |
| 当前结论 | HTTP 传输的授权建立在 OAuth 2.1 相关机制上；资源服务器必须验证令牌受众，不能把收到的令牌透传给下游；客户端动态注册、第三方授权和本地服务器仍需分别建模 |
| 适用范围 | 通过 MCP 连接企业工具、资源或远程服务的试点与生产系统 |
| 不代表什么 | 协议兼容不等于 Server 已获准；工具描述、返回内容和第三方 Server 都可能不可信 |
| 最小验证 | 固定协议版本；校验资源指示与受众；最小权限；读写工具分离；模拟令牌透传、混淆代理、SSRF、本地 Server 劫持与工具描述注入 |
| 生产证据 | Server 清单、负责人、来源与版本；授权流程；权限范围；工具 schema；出站策略；审计与撤销记录 |
| 重新核查条件 | 协议版本、授权方式、Server、工具权限、网络出口或身份提供方变化 |

官方来源：[MCP Authorization 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)、[MCP Security Best Practices](https://modelcontextprotocol.io/docs/tutorials/security/security_best_practices)。

## 技术卡 2：OpenTelemetry 生成式 AI 语义

| 字段 | 核查结论 |
|---|---|
| 对象 | OpenTelemetry Semantic Conventions 与生成式 AI 追踪 |
| 已核查版本 | Semantic Conventions `1.43.0`；GenAI 语义已迁移至独立仓库继续演进 |
| 当前结论 | 通用 trace/span 可用于串联请求，但生成式 AI 属性仍会变化；实现必须固定语义版本并保存内部字段映射 |
| 适用范围 | 模型、检索、工作流、工具和人审的试点与生产可观测 |
| 不代表什么 | 采用标准字段不自动产生端到端业务追踪，也不允许把敏感 Prompt 和输出默认写入日志 |
| 推荐映射 | 标准字段记录模型系统、操作、token 与响应；组织命名空间记录 `business_task_id`、`business_object_id`、知识快照、评估集、审批与后置条件 |
| 最小验证 | 跨服务 trace 传播；采样策略；敏感字段脱敏；失败工具与重试关联；业务结果回写；字段升级兼容测试 |
| 重新核查条件 | OTel/GenAI 语义发布、SDK 升级、模型供应商字段或组织日志政策变化 |

官方来源：[OpenTelemetry Semantic Conventions](https://opentelemetry.io/docs/specs/semconv/)、[GenAI semantic conventions repository](https://github.com/open-telemetry/semantic-conventions-genai)。

## 技术卡 3：本地推理容量测算

| 字段 | 核查结论 |
|---|---|
| 对象 | 部门级文本生成服务的容量与压测方法 |
| 已验证环境 | 本书公式与标准库样例；未绑定特定模型、量化格式或 GPU |
| 当前结论 | 单机可回答只能证明功能路径；生产判断还需要权重与 KV Cache 内存、并发、输入/输出长度、TTFT、生成速率、队列、背压、故障与回退证据 |
| 估算起点 | `所需生成吞吐 ≈ 峰值每秒到达任务 × 平均输出 token`；`并发中的任务 ≈ 到达率 × 平均响应时间`，二者都只能作为压测起点 |
| 样例 | 峰值 0.8 任务/秒、平均输出 450 token，则生成侧起点约 360 token/秒；平均端到端 12 秒时，在途任务约 9.6 个，仍需用真实长度分布和调度策略压测 |
| 必测分层 | 单请求质量；稳态并发；短时峰值；长上下文；故障与恢复；模型热切换；资源不足时的拒绝或降级 |
| 生产证据 | 硬件、驱动、运行时、模型与量化哈希；输入/输出分布；P50/P95/P99；错误与队列；功耗与成本；质量回归 |
| 重新核查条件 | 模型、量化、上下文、硬件、运行时、并发策略、服务目标或流量分布变化 |

容量公式用于设计实验，不是采购承诺。压测必须使用目标环境和近似真实请求分布。

## 技术卡 4：中国境内生成式 AI 与内容标识问题清单

| 字段 | 核查结论 |
|---|---|
| 对象 | 中国境内生成式 AI 服务的适用范围、治理责任与生成合成内容标识 |
| 已核查文件 | 《生成式人工智能服务管理暂行办法》；《人工智能生成合成内容标识办法》及配套强制性国家标准说明 |
| 当前结论 | 是否面向境内公众提供服务、服务类型、数据与内容流、生成内容传播方式会影响适用分析；标识办法自 2025-09-01 起施行 |
| 适用范围 | 在中国境内部署、采购或运营生成式 AI，尤其是面向公众提供生成内容的项目 |
| 不代表什么 | “内部使用”“本地部署”或“由供应商备案”都不能单独替代适用性判断；本卡不是法律意见 |
| 项目问题 | 谁是服务提供者；用户是否为公众；输入数据来源与个人信息如何处理；输出如何显式/隐式标识；日志、投诉、未成年人、内容安全和跨境链路由谁负责 |
| 生产证据 | 法务适用性结论、数据流图、供应商责任、标识实现与测试、内容处置、投诉响应、合同和备案/登记核查记录 |
| 重新核查条件 | 服务对象、地区、数据流、传播渠道、模型/供应商、产品能力或法律规则变化 |

官方来源：[《生成式人工智能服务管理暂行办法》](https://www.cac.gov.cn/2023-07/13/c_1690898327029107.htm)、[《人工智能生成合成内容标识办法》](https://www.cac.gov.cn/2025-03/14/c_1743654684782215.htm)。

## 技术卡 5：AI 风险与治理框架版本

| 字段 | 核查结论 |
|---|---|
| 对象 | NIST AI Risk Management Framework（AI RMF）与生成式 AI Profile |
| 已核查版本 | AI RMF 1.0；NIST AI 600-1 Generative AI Profile（2024） |
| 当前结论 | AI RMF 1.0 以 Govern、Map、Measure、Manage 组织风险管理；NIST 已公开说明 AI RMF 正在修订，因此项目引用必须保留版本 |
| 与本书关系 | 五道门、四平面和 A0—A4 是交付工具，可把证据映射到 RMF 功能，但二者不等价，也不构成 NIST 认证 |
| 最小验证 | 风险负责人、影响范围、测量方法、处置决定和持续监督都有证据；生成式 AI 特有风险进入评估和事故机制 |
| 重新核查条件 | NIST 发布新版 AI RMF/Profile，或组织采用新的管理体系与审计要求 |

官方来源：[NIST AI Risk Management Framework](https://www.nist.gov/itl/ai-risk-management-framework)、[NIST AI 600-1](https://nvlpubs.nist.gov/nistpubs/ai/NIST.AI.600-1.pdf)。
