# 附录 D：配套代码说明

本书的代码不是生产框架，而是把关键架构原则变成可运行示例。所有示例只使用 Python 标准库，便于读者理解、运行和修改。

代码目录：`code/`

## 1. 策略驱动的模型路由

文件：`code/model_router.py`

演示内容：

- 按数据级别和任务筛选允许渠道。
- 安全条件优先于成本和延迟。
- 没有合法路径时明确阻断。
- 记录路由原因，便于审计。

## 2. 权限感知的知识检索

文件：`code/permission_rag.py`

演示内容：

- 文档携带部门、项目和用户权限标签。
- 检索前先做授权过滤。
- 结果保留文档 ID 和引用。
- 无权限与无答案分别处理。

示例使用简单词项匹配，不代表真实向量检索质量。

## 3. 幂等工作流与人工确认

文件：`code/idempotent_workflow.py`

演示内容：

- 业务任务 ID 和幂等键。
- 草稿、待确认、结果未知、已执行和失败状态。
- 相同键与相同载荷复用结果；相同键与不同载荷明确冲突。
- 批准绑定业务对象与内容版本，旧批准不能用于新内容。
- 写回超时先读取业务后置条件，再决定成功或失败。

## 4. 评估与发布门禁

文件：`code/evaluation_gate.py`

演示内容：

- 阻断项与平均质量分分开。
- 质量、性能、成本和风险共同验收。
- P95 使用 nearest-rank 定义，避免小样本下标偏低。
- 输出通过、条件通过或阻断结论。

## 5. 端到端企业 AI 示例

文件：`code/enterprise_ai_pipeline.py`

这个示例把前四个独立机制连接为一条完整任务链：

```text
业务请求信封
-> 数据与能力路由
-> 用户权限过滤
-> 有引用的确定性草稿
-> 具体变更确认卡
-> 人工批准
-> 幂等写回
-> 可审计 trace
```

示例中的 `RequestEnvelope` 携带业务对象、用户、项目权限、数据级别和所需能力；`EnterpriseAIPipeline.prepare()` 先选择获准模型路径，再检索用户有权看到的证据。需要引用却没有获准证据时，任务明确阻断，不生成猜测性草稿。

准备完成后，系统返回包含业务对象和内容版本的确认卡，但不会自动写入。`approve_and_commit()` 只有得到绑定同一对象与版本的明确批准才执行写回；相同幂等键和相同载荷重复提交只产生一次业务动作，不同载荷会被拒绝。`audit()` 返回从请求、路由、检索、批准到写回的最小 trace。

代码使用确定性草稿函数，读者可以把它替换为自己的模型调用，但应保留路由、权限、证据、人审、幂等和追踪边界。直接把模型 SDK 填进示例，并不意味着获得了生产级模型网关、IAM 或可靠存储。

## 6. 完整演示

运行：

```bash
python3 code/run_demo.py
```

运行测试：

```bash
python3 -m unittest discover -s code/tests -v
```

当前测试集包含 16 个标准库单元测试。它不仅检查普通路径，还验证敏感请求选择本地通道、跨项目文档被过滤、无证据任务被阻断、幂等载荷冲突被拒绝、批准绑定内容版本、未知结果经过后置条件对账、小样本 P95，以及阻断项能够覆盖平均质量分。

## 推荐的阅读与改造顺序

1. 先运行测试和 `run_demo.py`，观察当前边界。
2. 修改数据级别或用户项目权限，确认路由与引用变化。
3. 添加一个会返回失败或 `OutcomeUnknown` 的写入函数，观察任务状态并实现后置条件查询。
4. 为自己的场景增加评估样本和阻断项。
5. 最后才替换模型、检索和存储实现，并为新增依赖补认证、版本、监控、容量与恢复。

## 使用边界

- 示例未连接真实模型、数据库、IAM 或企业系统。
- 示例要求 Python 3.10 或更高版本，只使用标准库；本版在 Python 3.12 环境完成测试。
- 生产实现需要替换内存存储、增加认证、密钥管理、日志最小化、持久化、并发控制和事故能力。
- 代码中的数据级别、阈值和策略只用于解释，不是企业安全或合规结论。
