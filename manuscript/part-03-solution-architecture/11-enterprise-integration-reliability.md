# 第 11 章 AI 如何可靠连接企业系统

演示环境里，智能体成功把一条记录写进 CRM。生产环境中，同一次调用在超时前其实已经写入，系统重试后又写了一遍。界面只显示“调用失败”，后台却多出了两条真实记录。

从 AI 开始改变真实记录的那一刻起，问题就不再只是“回答对不对”，还包括“这次动作到底有没有安全完成”。身份、权限、避免重复写入、事后核对和人工接管，都要在动手接系统前想清楚。

启明科技也遇到了同样的问题。销售方案已经生成，但保存草稿时接口超时。项目组不能让模型猜测结果，只能先查询客户系统，确认草稿是否存在，再决定要不要重试。

## 从会说到会做，风险发生了变化

知识问答回答错了，用户可能发现并纠正；系统写错 CRM、重复创建订单或误发客户邮件，影响会进入真实业务。工具调用让 AI 产生更大价值，也让权限、状态和恢复成为架构核心。

企业系统集成不是“给智能体几个工具”。它要定义连接方式、身份、动作契约、失败处理和审计证据。

企业系统通常通过五种方式连接。

| 方式 | 适用场景 | 优点 | 主要风险 |
|---|---|---|---|
| API | 系统提供稳定接口 | 边界清楚、可测试 | 鉴权、版本、限流、部分失败 |
| MCP | 为 AI 暴露结构化资源和工具 | 发现与调用方式统一 | 第三方信任、工具描述、权限委托 |
| 事件/消息 | 状态变化触发异步流程 | 解耦、可排队、适合长任务 | 重复、乱序、最终一致性 |
| 数据库/数据服务 | 受控查询或分析 | 数据直接、效率高 | 绕过业务权限和逻辑 |
| RPA | 无接口的旧系统 | 快速验证 | 脆弱、难审计、界面变化风险 |

选择连接方式要看系统能力和风险，不看概念是否新。能用稳定 API 的写操作，不应仅为了智能体体验改用屏幕自动化。

## 系统不能在调用途中忘记用户是谁

最危险的快捷方式之一，是所有用户共用一个高权限服务账号。应用知道用户是谁，但下游系统只看到超级账号，原有权限边界和审计责任消失。

更合理的方式包括：

- 用户委托令牌：下游按用户权限执行。
- 服务身份加细粒度授权：服务只能执行限定动作，并携带业务用户上下文。
- 临时授权：高风险动作在批准后获得短期权限。
- 只读与写入身份分离：读取不能自然升级为写入。

无论方式如何，日志都要能回答：哪名用户通过哪个应用，以什么服务身份，对哪个对象执行了什么动作，依据哪个批准。

## 每个工具都要说清楚自己能做什么

工具契约就像给外部人员办理一张临时放行条件卡。卡片不能只写“可以进公司”，而要限定哪扇门、什么时间、为了什么任务、能做哪些动作。任务结束后，权限也应随之失效。

工具名称“更新 CRM”太宽。一个可用契约至少包括：

- 用途和允许场景。
- 输入字段、类型、必填和取值范围。
- 调用者身份和所需权限。
- 读取、创建、更新、发送或删除哪类对象。
- 结果状态和错误码。
- 是否幂等。
- 超时和重试规则。
- 是否需要人审。
- 日志和敏感字段策略。
- 回滚或补偿方式。

模型可以帮助填参数，但不能改变契约。

工具契约最好使用机器可验证的结构，并为业务语义增加说明。字段 `status` 不是任意字符串，而是受控枚举。日期包含时区。金额同时包含币种。对象 ID 指明来自哪个系统。

可空字段说明“未知”和“无此项”的差别。模型输出先经过模式、范围和交叉字段校验，再进入工具适配层。

以 `create_proposal_draft` 为例。输入至少包括业务任务 ID、商机 ID、目标空间、标题、内容版本和审批上下文。返回值不只是文档 URL，还包括创建状态、文档 ID、最终权限、内容哈希和是否命中已有幂等记录。

这样，下游才能验证“创建了正确且可访问的草稿”，而不是只知道接口没有报错。

契约还要写清禁止行为。工具不能接受任意目标 URL，不能让模型覆盖负责人，也不能通过自由文本字段传入可执行脚本。下游系统的完整错误栈同样不能直接返回给最终用户。安全边界要体现在输入面，而不是完全依赖提示词提醒。

从“模型给出建议”到“企业系统发生变化”之间必须存在一条可验证执行链。用户身份和服务身份先限定可见对象，工具契约与业务规则再收窄输入。高影响变更停留在人审节点，幂等和后置条件负责处理重复执行与结果未知。补偿、人工接管和审计保证部分失败不会消失在一次接口返回里。

读取和写入最好分开。一个销售助手可能需要读取客户字段、创建方案草稿、更新商机状态和发送邮件，但不应把这些能力打包成一个“CRM 全能工具”。更稳妥的做法是拆成最小动作：

- `get_authorized_opportunity`
- `create_proposal_draft`
- `update_proposal_status`
- `request_price_approval`

试点期甚至不提供对外发送工具。工具越具体，权限、评估和审计越容易。

读操作也不是天然低风险。批量导出客户、跨项目搜索和读取薪酬数据即使不修改系统，也可能造成严重泄露。工具权限要同时限制对象范围、字段范围、查询数量、时间窗口和返回体积。智能体能调用“搜索 CRM”不代表它能枚举全库。

写操作则可以采用分阶段授权：先生成变更集，展示新旧值和影响对象。人或规则批准后，工作流获得短期执行令牌。令牌只能用于这次任务、这组字段和有限时间。批准的是具体变化，不是给智能体一次宽泛的长期权限。

## 失败重试，为什么会多写一条记录

模型、网络和外部系统可能超时。调用者不知道请求是否成功时，直接重试可能重复创建记录。

写操作应使用业务任务 ID 或幂等键。系统收到相同键时返回已有结果，而不是再次执行。无法原生幂等的系统，至少要在集成层检查重复对象并记录状态。

重试只适合暂时性故障。权限拒绝、参数错误和业务规则不满足不应自动重试，而应阻断或请求补充。

幂等键必须绑定业务意图，而不是每次网络请求随机生成。如果第一次调用超时，重试时换了新键，仍会重复创建。通常可以由“场景 + 业务对象 + 动作 + 业务版本”生成，或者在任务创建时一次性分配并贯穿所有重试。

还要区分“至少一次交付”和“恰好一次业务效果”。消息系统往往只能保证事件至少到达一次，消费者必须通过幂等记录、唯一约束或状态机避免重复效果。不要在架构图上写一个“消息队列”就认为重复问题已经解决。

对于异步写入，可以使用事务型发件箱：业务状态和待发送事件先在同一事务中保存，后台再可靠投递。消费端记录已处理事件。它无法消除所有分布式故障，但能避免“业务已更新、事件未发出”或“事件发出、业务状态未保存”的常见裂缝。

## 事情做到一半失败，怎样收场

多系统流程很难拥有传统数据库事务。方案文档创建成功、CRM 更新失败时，可以采用：

- 记录每个步骤状态。
- 保留已创建对象 ID。
- 从失败节点恢复。
- 必要时执行补偿，例如删除草稿或恢复状态。
- 将无法自动恢复的任务交给人工队列。

补偿不一定是完全撤销。例如已经发出的内部通知无法收回，可以追加更正并记录事件。

为长流程建立步骤账本会显著降低恢复难度。每一步记录期望前置状态、请求、结果对象、完成时间和补偿状态。恢复时先读取真实后置条件，而不是盲目重放全部步骤。模型可以协助解释失败，但不能自行修改账本来宣称任务完成。

补偿本身也是高影响动作，需要权限和失败处理。删除草稿可能失败，退款可能需要另一个审批，恢复旧值可能覆盖用户在故障期间的新修改。因此要区分自动补偿、人工确认补偿和不可逆动作。对不可逆动作，应把确认和最终检查放在执行前，而不是寄希望于事后回滚。

人工接管队列不能只显示“任务失败”。接管者至少需要业务对象、已完成步骤、失败原因、建议动作、已消耗预算、原审批和可用的恢复按钮。没有上下文的人工兜底，只是把系统复杂性转嫁给一线人员。

接口超时或限流时，还要有背压策略。

AI 任务通常比普通 API 更慢，而且可能同时调用多个下游系统。若每个节点都无限等待，少量慢请求就会占满工作线程和连接池；若所有失败立即重试，又可能形成重试风暴。

每条调用链需要分配总时限，再为检索、模型、工具和写回分配子预算。上游取消后，下游任务也应尽量取消；超过业务时限的结果即使后来成功，也不能悄悄写回。限流发生时，可以排队、降级为较小模型、减少非必要研究步骤或转为异步通知，但不能通过增加并发绕过供应商限制。

背压意味着系统主动放慢进入速度：队列达到阈值时拒绝新的低优先级任务，按部门配额调度，保留高价值任务容量。容量控制应与业务优先级一致，而不是让最会重试的客户端占满资源。

## 接入第三方工具时，多问一句

工具来自内部还是第三方，并不会改变基本原则：权限要在服务端重新检查，参数要经过校验，写操作要能确认结果。一个连接协议可以减少接入工作，却不会自动替企业承担这些责任。

生产环境还要定期核对本地任务和外部系统记录，找到那些“调用超时，但对方其实已经成功”的情况。第三方工具审查、协议细节和对账方法放在附录 H。

## 共享服务账号怎样让审计失去意义

某企业的内部助手使用一个管理员账号读取工单和客户记录。应用层日志保存用户姓名，因此团队认为审计完整。一次员工查询了不属于自己的客户，应用错误地返回相关记录。下游系统只看到管理员访问，无法判断原权限是否允许，也无法使用原系统的对象授权阻断。

事故后团队发现，应用日志可以被修改，管理员账号还被三个服务复用，无法可靠重建访问路径。修复不仅是补日志，而是引入用户委托或对象级授权代理，拆分只读和写入身份，限制服务权限，并在下游或可信适配层重新校验。

共享账号把权限控制从权威系统搬到一个更弱的应用中。它实现快，却会同时破坏最小权限、撤权、责任和事故调查。

AI 一旦开始调用工具，就进入了真实业务世界。这里没有“差不多成功”：一次写回要么能被确认和恢复，要么就可能留下重复记录和说不清的责任。
